Integrating Razorpay & Stripe in Django: The Real-World Guide

March 15, 2025 · Aniket Maithani · 8 min read

Why Dual Payment Gateways?¶

Every serious fintech product in India needs both Razorpay and Stripe. Razorpay for domestic UPI/NetBanking/Wallets, Stripe for international cards. Sounds simple until your webhook handler is getting duplicate events at 3am.

Here’s what the official docs won’t tell you.

The Setup¶

Start with a clean abstraction. Don’t scatter gateway-specific code across your views.

# payments/gateways/base.py
from abc import ABC, abstractmethod

class PaymentGateway(ABC):
    @abstractmethod
    def create_order(self, amount: int, currency: str, metadata: dict) -> dict:
        pass

    @abstractmethod
    def verify_signature(self, payload: dict) -> bool:
        pass

    @abstractmethod
    def refund(self, payment_id: str, amount: int) -> dict:
        pass

Now implement per gateway. The key insight: normalize the response shape so your business logic doesn’t care which gateway fired.

Webhook Idempotency — The Hard Part¶

This is where most implementations break. Razorpay retries webhooks up to 5 times. Stripe retries for 3 days. Without idempotency keys, you’ll double-credit accounts.

# payments/models.py
class WebhookEvent(models.Model):
    event_id = models.CharField(max_length=255, unique=True)  # gateway's event ID
    gateway = models.CharField(max_length=50)
    payload = models.JSONField()
    processed_at = models.DateTimeField(null=True)
    status = models.CharField(max_length=20, default='pending')

    class Meta:
        indexes = [models.Index(fields=['event_id', 'gateway'])]

Then in your webhook view:

def handle_webhook(request, gateway_name):
    event_id = extract_event_id(request, gateway_name)

    # Idempotency check
    event, created = WebhookEvent.objects.get_or_create(
        event_id=event_id,
        gateway=gateway_name,
        defaults={'payload': request.data}
    )

    if not created and event.status == 'processed':
        return Response({'status': 'already_processed'}, status=200)

    # Process and mark
    process_payment_event(event)
    event.status = 'processed'
    event.processed_at = timezone.now()
    event.save()

    return Response({'status': 'ok'}, status=200)

Failure Recovery with Celery¶

Never process payments synchronously in the webhook handler. Use Celery with exponential backoff.

@shared_task(bind=True, max_retries=5)
def process_payment_task(self, event_id: str):
    try:
        event = WebhookEvent.objects.get(id=event_id)
        # ... processing logic
    except Exception as exc:
        raise self.retry(exc=exc, countdown=2 ** self.request.retries * 60)

Production Checklist¶

  • Always verify signatures before touching your DB
  • Log raw payloads before parsing (debugging lifesaver)
  • Use select_for_update() on critical account balance updates
  • Separate webhook processing queues per gateway
  • Alert on retry exhaustion — those are real money events
